A worked example — how it nests together
In plain words: one Programme → many Intakes → many Projects. Approved projects wait in a shared pool until they are placed into an intake whose education level and dates match.

Reading it: 'University 2025' holds two intakes. 'Cyber Defence Dashboard' sits inside Jun–Dec 2025 because it is University level, Approved, and its internship dates fall within that intake's period. 'Robotics R&D' and 'Cloud Migration' are approved but still in the pool — not yet placed into an intake. A Polytechnic or JC project would never appear here, because its education level would not match this University programme.

Project Request
Linked by: education_level (+ year) overlap only — no FK, no dependency  Incidental
Incidental — not a real relationship. A Project Request is raised by education level and does not need a programme to exist (IO can raise any number of requests, no limit per PC or year). They merely share the same education_level + year. The matching that does real work happens downstream: an approved Project is attached to an Intake (which belongs to a programme) when education level matches and the project's internship period falls within the intake's period.

Application Logic — End to End
①
Project Request Flow
1
IO Admin creates a Project Request in the system, specifying the PC code, recipient (PC Head), CC (AD(P&C)), and the number of placements needed per education level (e.g. PC11: 3 Uni, 4 JC, 2 Poly).
2
PC Head receives the request (via system notification or email output). Disseminates to their team to consolidate project submissions.
3
AD(P&C) submits projects into the system — either manually (one by one) or via Excel upload. Each project includes education level, internship period, duration, placement count, mentor info, and matching tags.
4
IO Admin reviews submitted projects and approves or rejects each one. System automatically tracks fulfilment: compares total submitted placement count vs requested count per PC + education level.
⚠️ System flags Shortfall or Over-allocation automatically.
②
Programme & Intake Setup (order-independent with Flow ①)
1
IO / IO Admin creates a Programme by selecting education level, adding a title and optional description. Year is auto-filled. Multiple programmes can be created for the same education level and year — no restriction.
2
IO / IO Admin adds Intakes to the programme. Each intake has a title (auto-generated), internship period (month/year), and application open/close dates. Multiple intakes per programme allowed; periods can overlap.
3
IO / IO Admin optionally attaches approved projects to an intake. The system shows only eligible projects: matching education level, Approved status, and internship period falls within intake period.
This step is optional — can be done later from the intake page or from a project's own page.
4
IO Admin sets the intake status to Open when ready to receive applicants. Status: Draft → Open → Closed → Completed.
③
Applicant Flow
1
Applicant views open intakes matching their education level. Application window must be active (between application_open and application_close dates).
2
Applicant submits application with personal info, academic background, and optional project preferences (ranked list from projects available in that intake).
3
IO / IO Admin reviews applications and shortlists candidates. Makes offers to shortlisted applicants.
4
Applicant accepts or rejects the offer. On acceptance, project is marked Assigned and applicant status updates to Accepted.
⚡
Key System Rules
  • Programme and Project flows are order-independent — IO Admin can create programmes or approve projects in any order; they are linked at intake setup time.
  • A project can only be attached to an intake if: education_level matches · review_status = Approved · internship period falls within intake period.
  • Project has a hosting period (window mentor is available) and a separate duration (how long the intern stays) — duration is manually entered, not calculated from the period.
  • Project Request fulfilment is tracked automatically: placements_submitted (sum of project headcounts per PC + edu level) vs placements_requested. Flags Shortfall or Over-allocation.
  • AD(P&C) can submit projects via manual form or Excel upload. Excel field names must exactly match the form. Multi-select fields use comma-separated values.
  • Applicants only see intakes where education_level matches and application window is open.
